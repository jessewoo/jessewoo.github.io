The Process

In 2013, the entire RCSB PDB team discussed our mission statement of being a premiere scientific site focused on usability and being cutting edge in delivering quality data and advanced tools. Consequently, we came to a consensus that the site needed to be redesigned - starting with the homepage. Not just to keep up with current design trends, but to do the following on the most visited page:

  • Reduce information overload and redundancy
  • Organize and group related content
  • Catch all different user groups from high school students to scientists
  • Think through user flow and goals, making it easier to accomplish tasks
  • Usher in new yet proven web technology to handle expanding scientific content
  • Able to be mobile and tablet responsive
  • Addressing user needs: make core features and tool more visible

The last point was a road block in the brainstorming phase of the homepage redesign. Was there a solution of making core features and tools accessible to users? Many users complained that they weren't even aware that such tools existed. Eventually after a month of deliberation, in January 2014 with an epiphany, I suggested a two-tier panel section similar in "design" of a vertical navigation with static dropdown menus. After presenting this solution to the leadership team, the site was "green-light" to proceed with code development.

After much discussions, team meetings, whiteboard scribbles, prototyping, and hundred of hours of code development, review and feedback, and testing, the redesigned homepage became a reality.

As the lead front-end developer, I personally took charge of this monumental project, which met expectations and deadlines. Once released, the redesiged RCSB PDB homepage was well received by the scientific communnity and team members. And analytics have shown a considerable lift in using our search tools, which was previously hidden.

With the redesigned framework (style guide, Twitter Bootstrap) set, subsequent pages like the Structure Summary page (2015) and the Search Results page (2016) was given a facelift, information was organized, and various user interactions were tightened down.

The homepage took about 10 months to complete from conception to production. The Structure Summary page was likewise the same. Learning from the previous releases, we wanted to give users the perspective that we are always continually developing our site, so we opted for a more agile development cycle with 5 week sprint phases. For the development of the Search Results page, I pushed for more and frequent releases similar to how Google or Facebook operates. Definitely, the paradigm of development has changed and been transformed.

Search Results Page Process

The Search Results page was overhauled with a phased approach. There were many things we wanted to do as a team, we wanted to break away from having to use Java beans for the transfer of data across the Java application but utilize a RESTful API approach, then there was redesigning the whole user experience. First phase would be to rehaul the UX of the site, that was simply putting into place our foundational design layout and utilizing Bootstrap to make HTML elements uniform, which took only 2 months to accomplish.

The second phase would be more technical, getting into the Java code and utilizing existing functions and classes to create for myself a JSON payload that the frontend code can utilize. Not familiar with Java, I had to learn on my own and asked my backend developer office mates on what is a main(String[] arg) or what was public class --- extends AbstractHibernateAction, but in the end I was able to produce the JSON payload on my own utilizing the data retrieved with Hibernate and the Data Access Object (DAO) layer.

Once the JSON payload was created, work needed to be done with page interactions. The JS code would do an AJAX call to the RESTful API, an internal function, everytime a user clicks on a refinement link, a sort option, or selected how many results per page. Previously before, it was a entire refresh of the page itself, but instead, we would just be updating the search results section with a nanobar indicating progress. The user would get the impression that it was faster as only that specific search results section was changing, everything else would stay the same. This was a milestone, it's an architecture that would be the building block of modularizing the site, using microservices.

Reasons and Beginnings

Our Classic Site was Difficult to Maintain

Built with outdated technology and years of different teams of developers working on the Java code base, it became clear over the years that it would be difficult to maintain and manage. The cost of keeping our classic site and the upkeep outweight the cost of switching over to a new architecture and paradigm. There were bandages on top of bandages, a lot of dead or redundant code, copy and pasted code littered everywhere, the start up time for a local Java build took minutes - that's an eternity for developers, too many non actionable error handling, and a mixture of frontend and backend code mashed into one monolithic code base where the separation of development task was difficult to assign. Overall, developers feared the code base, the seemingly black box code base. The code dated 10 years prior.

We were determined to bring order into the chaos, bring clarity into the opaqueness of the code base. This was our problem that we wanted to solve for our future success.

Gathering Inspiration and Going to the Drawing Board

Reading up on various tech articles, many well known companies like Netflix and Amazon moved their monolithic site to a microservice based architecture, which really inspired us to think similar lines. The advantage was speed, scale and agility, three things that were important to us for the future. Drilling down, if one of microservice had an outage, it wouldn't take down the entire site but the site would be able to handle such an error. Also, with the usage of services, the main code base wouldn't be bloated and large. Each change in the code base wouldn't be require a rebuild, redeployment that could take about 15 minutes on the Java framework.

In addition, there was Walmart who started using NodeJS for end to end usage of JS, which was inspiration from the best.

So taking from top notch companies, we wanted to move our code base in the same direction. As a team, RCSB PDB would start fresh and move to a new microservice-based architecture to better scale our services to accommodate growth and user demand, and to increase the speed at which we can deploy new services. And there were so many benefits even from the user.

This RCSB PDB article explains more in depth on what was done. There has been much optimization since the launch, but days before the launch, the load time on the Classic site was ~700ms vs the new NodeJS site of 425ms, huge reduction. It was a great win for the team.

Technology Used and Learnings

  • NodeJS uses Javascript on the server, JS being a language all frontend team members understood. NodeJS elminates waiting for a process to finish, uses callbacks. NodeJS does non-blocking I/O, which can be summed up in a fast food analogy.
  • Express was used for URL routing, application level and error handling middleware, and the incorporation of the Pug template engines.
  • Pug templates had build in iteration functions, mixin which create reusable code blocks, and conditional syntax to handle various situations.
  • Promises, which is basically an object representing the eventual completion of a asynchronous operation and it's resulting value. Promises were used to retrieve data from RESTful calls, based on success (.then) or failure (catch), the app would route to a rendered page or an error catching mechanism.
  • ES6 related syntax - template string, arrows, 'this', import and export of JS functions, usage of 'let' (varied) and 'const'
  • Currently using ReactJS (with axios, React-dom) to build an autocomplete suggestion box when users use the search bar, and with the Search Results page, going to utilize Redux - a single source to store state.

Learned from Front End Team Lead

  • Intersection Observer, defers the load of offscreen images in order to speed up the site. Polyfill needed for IE.
  • Winston easy to digest logs that could be transferred to another external log server to be analyzed.
  • memory-cache - enable high delivery speed of content based on cache key
  • passportJS - simple authentication for logging in.
  • child process - running simple scripts on server
  • Multiple configuration files based on server purpose - local, dev and producton
  • Chrome Network Tab to check reasons behind page load time, Google Chrome Audit tool, and Pingdom

Other JS libraries used

  • ChartJS - used for simple dynamic charts for statistics
  • d3.js - used for sunburst pie chart for statistics
  • DataTable.Net - used to show statistics table of data
  • ESlint - finding problemmatic issues with coding
  • Mocha and Chai and - unit test framework with assertion library
  • Bootstrap 3 - front end design library
